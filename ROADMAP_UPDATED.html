Roadmap עדכני ל-Gaming Vibe Coding

המטרה של ה-roadmap הזה היא לבנות את המוצר בצורה הכי יציבה, סקיילבילית ומעשית: קודם מנוע מוצרי חזק, אחר כך שכבת AI חכמה, ורק אחר כך Agents מתקדמים.

Engine First JSON Pipeline 2D + 3D Runtime Agents Later Product Before Magic

החלטת ליבה: לא לבנות קודם “מערכת Agents”, אלא לבנות קודם Game Generation Engine ברור ויציב. אחר כך מוסיפים Agents כשכבת שכל מעל מנוע שעובד.

למה זה הכיוון הנכון

מנוע קודם

אם אין מנוע ברור, כל פלט של AI יהיה אקראי, לא עקבי וקשה מאוד לדבג.

Agents אחר כך

Agents טובים כשהם עובדים מול חוזה ברור: schema, runtime, validation ו-template builder מסודר.

מוצר לפני טריק

המשתמש צריך לקבל משחק עובד תוך דקות. זה יותר חשוב מ-orchestration מרשים אבל שביר.

הארכיטקטורה המומלצת

1. Core Engine
  • Schema אחיד למשחקים
  • Runtime ל-Phaser / Three / Rapier
  • Template Builder
  • Validation
2. Product Layer
  • Auth
  • Tokens
  • Game storage
  • Preview / Download / Edit
3. Agent Layer
  • Intent Agent
  • MCQ Agent
  • Design Agent
  • Patch / Validation Agent

השלבים לפי סדר נכון

1

לסגור את ה-Core Engine

זה הבסיס של כל המוצר. בלי זה כל השאר נשען על חול.
  • להחליט סופית על מבנה JSON אחיד ל-2D ול-3D.
  • להגדיר אילו שדות הם חובה ואילו מודולריים.
  • לפרק מכניקות למשבצות קבועות: movement, combat, collectibles, ui, win/lose.
  • להפריד בין מה שה-AI מחליט לבין מה שה-runtime מריץ.
התוצאה הרצויה: כל משחק הוא וריאציה על מנוע ברור, לא קובץ כאוטי שמנסה להמציא את עצמו מחדש.
2

לבנות Runtime אמיתי ל-2D ול-3D

כאן המוצר הופך מ-JSON יפה למשחק שעובד באמת.
  • ל-2D: לבנות pipeline מסודר ב-Phaser.js.
  • ל-3D: לבנות pipeline מסודר ב-Three.js.
  • להכניס Rapier רק איפה שהוא באמת מוסיף ערך, בעיקר ב-3D ובפיזיקה מורכבת.
  • לשמור renderer פשוט, צפוי וקל לדבוג.
לא לנסות לתמוך “בכל סוג משחק בעולם” מהיום הראשון. לבחור subset ברור ולעשות אותו מצוין.
3

לעבור ל-Template Modules

במקום שכל משחק ייווצר מאפס, בונים ספרייה של אבני בניין.
  • מודול תנועה.
  • מודול מצלמה.
  • מודול אויבים.
  • מודול איסוף פריטים.
  • מודול inventory / quest / dialogue למשחקים שצריכים את זה.
זה מה שייתן לך יציבות. ה-AI לא יכתוב “משחק”, אלא ירכיב מנוע קיים בצורה חכמה.
4

לשפר את שכבת ה-Product

כאן אתה דואג שהמוצר יהיה שימושי, לא רק טכנולוגי.
  • Game dashboard מסודר.
  • Better preview.
  • Download package מסודר יותר.
  • Asset pipeline בסיסי.
  • Error states ידידותיים למשתמש.
המשתמש לא אמור להבין מה זה JSON או engine. הוא אמור להרגיש שהמערכת “מבינה משחקים” ויודעת לבנות לו אותם.
5

רק עכשיו להוסיף Agents

אחרי שיש מנוע יציב, agents נהיים מכפיל כוח במקום מקור לבלגן.
Intent Agent

מבין מה המשתמש באמת רוצה לבנות.

MCQ Agent

מחליט אילו שאלות באמת חסרות כדי להמשיך.

Design Agent

מנסח game spec מסודר לפני JSON.

JSON Author Agent

מייצר JSON לפי schema ולא “ממציא מנוע”.

Patch Agent

עורך משחק קיים בלי לשבור את זהותו.

Validation Agent

בודק חורים, חוסר איזון, ושדות חסרים.

6

להכניס Asset Pipeline אמיתי

רק אחרי שה-gameplay יציב, מוסיפים שכבת ויז'ואל וסאונד רצינית.
  • Sprites / textures / sound fx / music.
  • Placeholder assets כברירת מחדל.
  • Generated assets כאופציה, לא כתלות חובה.
  • Manifest מסודר לכל asset.
7

לשפר Quality Loop

כאן המערכת מתחילה להיות באמת חכמה ולא רק “מגיבה”.
  • Playtest checks.
  • Balance checks.
  • Schema linting.
  • Regression templates לפי genre.
  • Retry / correction loops יותר חכמים.

מה הכי כדאי לבנות עכשיו, ממש בשלב הנוכחי

השבוע הקרוב
לסגור את ה-runtime של 2D כמו שצריך עם Phaser, כולל 3-4 genres שעובדים טוב באמת.
אחרי זה
לבנות builder מודולרי: שה-AI בוחר מודולים במקום לנסות “לכתוב משחק” מאפס.
שלב הבא
להרים 3D בסיסי יציב עם Three.js ולחבר Rapier רק למכניקות שצריכות פיזיקה אמיתית.
רק אז
להתחיל לפרק את שכבת ה-AI ל-agents נפרדים.

מה לא כדאי לעשות מוקדם מדי

  • לא לבנות orchestration מסובך של 8 agents לפני שיש engine יציב.
  • לא לתמוך בכל genre בעולם בבת אחת.
  • לא לתת ל-AI לייצר runtime code חופשי בלי גבולות.
  • לא להפוך assets generation לתלות חובה ביצירת משחק.
  • לא לקשור את כל המוצר ב-provider AI אחד בלי fallback.

במשפט אחד

הדרך הכי טובה לבנות את הפרויקט הזה היא: Engine קודם, Templates אחר כך, Product layer יציב, ורק אז Agents.

נוצר כ-roadmap עדכני לפרויקט Gaming Vibe Coding. המטרה: סדר, עדיפויות נכונות, ופחות סיכוי לבנות מערכת מרשימה אבל שבירה.
