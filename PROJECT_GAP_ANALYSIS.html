3 Backend gaps

The backend is the most mature part of the codebase. The remaining gaps are about productionization rather than features.

Auth & account lifecycle

  • Critical No password reset / forgot password
    If a user forgets their password they are locked out forever. There is no email transport configured.
    Fix: add /api/auth/request-reset + /api/auth/reset, email via Resend / Postmark / SES, signed token with 15-min expiry.
  • High No email verification
    Anyone can register with any email. This becomes a problem when you start spending OpenAI tokens.
    Fix: verification email on register, block spend until verified.
  • High No account settings endpoints
    User cannot change display name, password, or email. There is no PATCH /api/auth/me.
  • Medium No OAuth providers
    Spec mentioned "Google/Discord TBD". Non-technical users on a SaaS expect Google sign-in.
  • Medium No JWT refresh / revocation
    Token never expires server-side. A leaked token is valid forever. Add a token version column or short-lived JWT + refresh token.

Tokens, billing, abuse

  • Critical Tokens cannot be purchased
    There is a token balance, a grant-by-admin route, and per-action spending — but no Stripe customer, no checkout, no webhook, no plan upgrade endpoint.
    Fix: Stripe Checkout + /api/billing/portal, webhook to credit tokens, idempotency keys, store invoices.
  • High No daily / per-IP cost cap
    A bug or a malicious user could burn the OpenAI budget. Rate limiting only protects request count, not cost.
    Fix: daily $ cap per user and global cap, log usage.total_tokens per call, kill switch when threshold exceeded.
  • Medium Token cost constants are static
    TOKEN_COSTS.NEW_GAME etc. live in code. Pricing changes require a deploy.
    Fix: move into a config table or env-driven JSON.
  • Medium No refund / failure policy in code
    Spec says "no refund on failure" — but token spend currently happens before generation. If OpenAI fails after the spend, the user lost tokens for nothing.
    Fix: two-phase reservation (hold tokens, commit on success, release on hard failure). At minimum, document this clearly in the UI.

Generation pipeline

  • High Generation is synchronous
    A request to /api/generate-game can take 30-90 seconds. The HTTP connection ties up a worker, and any client retry causes a duplicate spend. There is no job queue, no progress streaming.
    Fix: background queue (BullMQ + Redis, or simpler: in-process worker with Server-Sent Events) and a POST /api/jobs + GET /api/jobs/:id/stream pattern.
  • High No streaming of model output
    Users wait staring at a spinner for >30s. Even a fake "AI is thinking…" stream improves perceived performance.
  • Medium No prompt-injection protection
    The user prompt is concatenated straight into the system prompt. Hostile input can hijack the schema or exfiltrate the system prompt.
    Fix: wrap user input in clearly delimited <user_request>...</user_request>, instruct the model to ignore instructions inside it, validate the output Zod schema strictly (already done — keep doing it).
  • Medium Dual JSON shape is confusing
    Old gameJSON (legacy templateBuilder) and new GameDefinition (engine) are both alive. The frontend currently uses the legacy path. Pick one as canonical and have the other auto-translate.
  • Low Fallback generator is a band-aid
    fallbackAIService is great for dev when OpenAI is down, but in production it produces a template game that does not match the user's prompt. Consider disabling in NODE_ENV=production.

Data / persistence

  • High SQLite is single-instance
    You cannot horizontally scale. Backups are file-copies. WAL files exist (gaming-vibe.db-wal) so a careless deploy could corrupt data.
    Fix path A: stay on SQLite + Litestream replication (cheap, fine for early users). Fix path B: migrate to Postgres before public launch.
  • High No automated backups
    No nightly dump, no off-host backup. Losing the SQLite file means losing every user game.
  • Medium No soft-delete / trash
    DELETE /api/games/:id is hard-delete. A non-technical user clicking by mistake loses work.
  • Medium No game version history
    Spec says "no history" but most users will eventually ask "undo my last edit". A 5-snapshot ring buffer is cheap.
  • Medium Asset manifest exists but assets are not stored
    buildAssetManifest() records URLs but the backend does not host or proxy generated assets. Generated images would need an object store.

Other backend gaps

  • High No GET /api/health deep readiness
    Current health endpoint is good but does not verify OpenAI connectivity. A failing key still shows green until the user generates.
  • Medium No CORS allow-list
    Currently relies on cors() defaults. Production should pin allowed origins per env.
  • Medium No request/response size limits documented
    Express defaults silently accept multi-MB JSON. Set explicit express.json({ limit: '256kb' }) with bigger limit only on game-create.
  • Medium No structured error codes
    Errors return free-text messages. Frontend would benefit from stable error codes (OPENAI_NOT_CONFIGURED, TOKENS_INSUFFICIENT, …).
  • Low No OpenAPI / generated SDK
    Zod schemas are server-only. Frontend types are hand-maintained — they will drift. Consider zod-to-openapi or shared @gvc/contracts package.

4 Frontend gaps (the new app)

The new frontend/ covers the documented MVP flow end-to-end. Below is everything that is intentionally not yet wired and should land before public launch.

Functional gaps

  • High Engine generation path is implemented in API but not exposed in UI
    engineApi.generate() exists in src/api/endpoints.ts but no page calls it. Users cannot try the new GameDefinition runtime from the product.
    Fix: add a "Use new engine (beta)" toggle on the Builder page; render the result via an in-page GameRuntime import from gvc-game-engine, or a dedicated /build/engine route.
  • High No real onboarding
    First-time user lands on dashboard with no tutorial. Non-technical users will not know what to type as a prompt.
    Fix: empty-state dashboard with 3 example prompts, "first game in 60 seconds" guided flow, prompt suggestions on the Builder page.
  • High No password reset / change-password UI
    Even when the backend ships it, the frontend has no /forgot or /account route.
  • High No upgrade / billing screen
    Token badge shows balance but there is no way to buy more.
  • Medium No game search / filter / sort
    Games list shows everything in a flat list. Becomes painful past ~20 games.
  • Medium No share-link / public preview
    Backend has isPublished flag; frontend ignores it. There is no public "view someone else's game" route.
  • Medium No auto-refresh of token balance
    Token badge refreshes only after generation calls. If admin grants tokens out-of-band, the UI lies until the user clicks the badge.
  • Medium No undo for delete
    A confirm() dialog is the only safety net. A toast with "Undo" + 5-second window would be friendlier.
  • Medium Edit history is invisible
    After several edits the user has no record of what they asked. Surface the prompts log (prompts_history table) in the GameDetail page.
  • Low No keyboard shortcuts in builder
    Ctrl+Enter to generate, / to focus prompt — small wins.

Engineering gaps

  • High No tests
    Zero unit, component, or e2e tests in frontend/. The whole client could silently regress.
    Fix: Vitest + React Testing Library for hooks and critical components (AuthContext, GameBuilder flow). Playwright for one happy-path e2e.
  • High No global toast / notification system
    Errors are shown inline per page. Background actions (delete, download) use alert(). That is not SaaS-grade.
  • High No data layer caching
    Every navigation refetches games. React Query would be a one-day win — caching, retries, deduping, stale-while-revalidate, optimistic updates.
  • Medium Iframe sandbox is permissive
    GamePreview uses allow-scripts allow-same-origin. Same-origin lets the embedded HTML read parent storage. Drop allow-same-origin and host previews on a separate origin (e.g. preview.gvc.app) before public launch.
  • Medium No error boundary
    A render error in any route blanks the whole app. Add a top-level <ErrorBoundary> with a "Reload" CTA.
  • Medium No analytics
    No PostHog / Plausible / GA. We will not know what users actually do.
  • Medium No theming / dark mode
    Spec said "not dark dashboard" — fine. But we should still respect prefers-color-scheme for the marketing pages later.
  • Medium No i18n
    Project author works in Hebrew. The product currently ships English only. At minimum extract strings and add an en/he.json with react-i18next.
  • Medium No accessibility audit
    No aria-live for progress messages, color-only state indicators, no focus-trap on the health-popover.
  • Low Bundle is one chunk
    The whole app is one ~205KB JS file. Lazy-load StatsPage, GamePreview, and engine-runtime when added.
  • Low No PWA
    No service worker, no installability, no offline fallback page.

5 GAME_ENGINE gaps

The engine has a real ECS architecture and supports 2D + 3D + physics + tweens + spawners + UI. The biggest gaps are integration and DX.

  • Critical Engine is not consumed by the new frontend
    The frontend uses backend-rendered htmlString in an iframe. The engine therefore only runs in GAME_ENGINE/examples/playground. Two parallel runtimes is technical debt.
    Fix: add the engine as a workspace dependency in frontend/package.json, expose a <EngineCanvas definition={…} /> component, and route engine-generated games to it.
  • High No mobile / touch input bindings
    VirtualJoystick exists but the GameDefinition schema does not let the AI specify touch controls per genre. Phones cannot play the generated games.
  • High No asset loading from arbitrary URLs in safe mode
    Sample games reference local ./assets/.... There is no CDN-prefix awareness, no signed URL handling, no error UI when an image 404s.
  • Medium No save / load game state in-engine
    GameStateStore has snapshots but no persistence. Users cannot resume a game after refresh.
  • Medium No FPS / performance budget guard
    Generated content can spawn 1,000 entities and tank low-end devices. Add an entity-cap warning + a low-spec mode toggle.
  • Medium No audio assets pipeline
    AudioSystem exists but the AI is never told to add SFX/music, and no royalty-free pack is wired in.
  • Medium Engine bundle size will be heavy
    Three.js + Phaser + Rapier in one bundle is ~1.5MB+ gzipped. When integrated, code-split per renderer (only load Three.js for 3D games, only Phaser for 2D).
  • Low No editor / inspector overlay
    A devtools-like panel showing entity tree, state values, scene graph would help during prompt iteration.
  • Low Missing high-level capabilities the AI does not know about
    No particle system, no post-processing, no shaders, no networking, no save-slot helpers. Either add to the engine, or document them as out-of-scope.

6 Cross-cutting concerns

Repo & tooling

  • High No CI
    No GitHub Actions / GitLab CI. Tests, type-checks, and lint do not run on PRs.
  • High No shared contracts package
    Backend Zod schemas and frontend TS types are duplicated. They will drift.
  • Medium Frontend is not in the npm workspaces list
    Root package.json only lists GAME_ENGINE, prototype, prototype/backend. Add frontend so installs are unified.
  • Medium No formatter/linter
    No Prettier config, no ESLint config in the new frontend. Inconsistent style will appear with multiple contributors.
  • Low No .editorconfig / commit hooks

UX consistency

  • High RTL/Hebrew support unverified
    Author writes in Hebrew. UI is LTR-only. CSS does not use logical properties.
  • Medium Mixed iconography
    Sidebar uses Unicode glyphs (⌂, ✨). Replace with a real icon set (Lucide / Phosphor) for crisper rendering.
  • Medium No empty / error / loading skeletons
    Lists show a spinner. Skeletons feel faster.
  • Low No motion design
    Page transitions and progress steps are static. A small framer-motion pass would lift perceived quality.

7 Security & privacy

  • Critical Generated HTML runs with allow-same-origin
    The model literally writes JavaScript that runs in the user's browser. With allow-same-origin, that script can read localStorage (auth tokens!) on a successful XSS / prompt-injection.
    Fix: drop allow-same-origin; serve previews from a sibling origin (e.g. preview.gvc.app); add CSP default-src 'self' to the generated HTML; review what templateBuilder emits for unescaped user content.
  • High No CSP on the main app
    Helmet is on, but no explicit Content-Security-Policy. With AI-generated content nearby, a hardened CSP is mandatory.
  • High No audit log
    Analytics table tracks events but admin actions (token grants, bans) are not separately logged. For compliance and abuse triage you want a tamper-evident trail.
  • Medium No GDPR / privacy basics
    No cookie banner (we use localStorage, not strictly cookies, but still), no privacy policy, no "delete my account" endpoint, no data export.
  • Medium No abuse / content moderation
    Users can prompt for hateful or copyrighted content. Add OpenAI moderation (omni-moderation-latest) on every prompt before generation.
  • Medium Secrets handling
    No example deploy script for rotating OPENAI_API_KEY, no secret manager integration, only .env files.

8 Observability & ops

  • High No external error reporting
    Pino logs to disk only. Sentry / Highlight on both backend and frontend would catch real-world failures.
  • High No cost dashboard
    You currently cannot answer "how much OpenAI spend per active user this week?" without reading SQLite by hand.
  • Medium No metrics export
    No Prometheus, no OpenTelemetry. /api/stats is admin-only and aggregated; you cannot alert on it.
  • Medium No uptime monitoring
    No external pinger. Combine with structured /api/health readiness checks.
  • Medium No deploy artifact / Dockerfile
    There is no canonical way to ship the backend or the frontend. start.bat exists for local dev only.

9 Monetization & payments

  • Critical No payment provider integrated
    Plans are described in docs but the only way to get tokens is the admin grant route. The product cannot earn money.
    Fix: Stripe (Checkout for one-shot token packs, Billing for subscriptions). Webhook → credit tokens with idempotency. Customer portal for cancel / update card. Tax handling via Stripe Tax. Receipts via email.
  • High No plans table / feature gating
    subscriptionTier is a free-text column with no enforcement matrix. Define plans (Free / Pro / Studio) and codify which features each can use.
  • Medium No referral / invite credit
  • Medium No coupon / promo code support

10 Asset generation pipeline

Both tools/game-dev-pack/asset-generation/fal-unverified and retro-unverified exist but are not connected to any code path. Generated games today look like colored boxes.

  • High No image generation in the loop
    A platformer "with a fox" still gets a beige cube. fal.ai / Replicate / DALL·E should generate sprites and the engine should consume them.
    Fix: after the GameDefinition is produced, run a second AI pass to enumerate needed assets (player, enemies, backgrounds), generate them, store in object storage, and inject URLs into the definition's assets array.
  • High No object storage
    There is no S3 / R2 bucket configured. Generated images would have nowhere to live.
  • Medium No royalty-free fallback pack
    When generation fails, the engine should fall back to a curated kit of CC0 sprites and audio.
  • Medium No audio generation
    No SFX/music pipeline at all (e.g. ElevenLabs sound, Suno).

11 Testing & quality

  • High One backend test file, no frontend tests, sparse engine tests
    prototype/backend/tests/backend.test.js exists. GAME_ENGINE/tests/ has files (good). frontend/ has none.
  • High No e2e
    A Playwright run covering "register → set API key in admin → generate game → see preview" would catch 80% of regressions.
  • Medium No load test
    k6 or Artillery to verify the rate limiter and SQLite under concurrent generation requests.
  • Medium No visual regression
    Chromatic / Percy on the builder + dashboard.
  • Medium No prompt regression suite
    Pin a small set of "golden prompts" and snapshot the resulting GameDefinition. Detect when a model upgrade silently changes outputs.

13 Recommended new systems

Beyond fixing what is already there, these are systems I would actively add. Each one moves the product noticeably closer to a real SaaS.

1. Job queue + progress stream

Move generation off the request thread. Let the Builder UI show real progress (tokens used, model thinking, validating, building HTML).

2. Shared @gvc/contracts package

One Zod source of truth, consumed by backend and frontend. Compile-time guarantee that types match.

3. Asset Studio

Per-game asset gallery with regenerate / replace / upload. The user can swap a sprite without re-running the whole generation.

4. Prompt Library

Curated prompt templates (RPG, runner, puzzle…) the user can pick and tweak. Doubles as marketing collateral.

5. Public Gallery

If isPublished, show the game on a public discover page. Users sharing their creations is the cheapest growth loop.

6. Remix flow

"Open this public game in my account and edit it." Forks are powerful and well-understood by non-technical users (Scratch, CodePen).

7. Versioned snapshots

Even if "no history" is the policy, a 5-step ring buffer per game is cheap and answers "undo" requests.

8. Soft-delete + Trash

30-day trash. Avoid panicked support tickets when someone clicks Delete.

9. Moderation gate

OpenAI omni-moderation-latest on every prompt. Block hate / sexual minors / violence-against-real-person before spending tokens.

10. Cost guardrails

Per-user daily $ cap; global kill switch. Prevents a single bug or abuser from draining the OpenAI budget.

11. Admin console

The current /stats page is a start. Add user search, ban, refund-tokens, view recent generations, replay prompt.

12. Telemetry & product analytics

PostHog. Funnels: signup → first prompt → first generated game → second edit. We need to know where users drop.

13. Embed SDK

A <script src="…/embed.js"> that lets users embed their generated games on their own sites. Strong viral mechanic.

14. CLI / API tier

A future "Studio" plan that exposes the JSON API to power users — they can integrate game generation into their own pipeline.

14 Suggested roadmap

Concrete, ordered, and small enough to actually finish. Each phase is ~1–2 weeks of focused solo work.

1 Stabilize MVP

Goal: real users can sign up, generate, play, save without losing data.
  • Sentry on backend + frontend; structured error codes.
  • SQLite backups (Litestream to S3) or Postgres migration.
  • Two-phase token reservation (hold / commit / release).
  • Iframe sandbox hardening: drop allow-same-origin, separate preview origin, CSP on generated HTML.
  • Password reset flow (backend + frontend).
  • Global toast system + error boundary in frontend.
  • Frontend tests for Auth + Builder happy path; one Playwright e2e.

2 Make it earn money

Goal: a stranger can buy tokens with a credit card.
  • Stripe Checkout + webhook + customer portal.
  • Plans table + feature gating matrix.
  • Billing page in frontend.
  • Daily $ cap per user + global kill switch.
  • Email transport (Resend) for receipts, verification, reset.

3 Make the games actually good

Goal: generated games stop looking like colored boxes.
  • Wire tools/game-dev-pack into the pipeline — image generation per game.
  • Object storage (S3 / R2) for assets.
  • Royalty-free fallback pack.
  • Switch frontend preview to engine runtime for new games (GameDefinition path).
  • Mobile / touch input bindings in engine; test on phone.
  • Audio generation (SFX + ambient music).

4 Make it grow

Goal: users invite users.
  • Public game gallery + share links.
  • "Remix this game" flow.
  • Embed SDK.
  • Onboarding tour, prompt library, examples.
  • Referral credit.
  • PostHog analytics; funnels for activation.

5 Productionize

Goal: it runs without you watching it.
  • Dockerfiles + docker-compose.
  • GitHub Actions CI: typecheck, test, build for backend, engine, frontend.
  • Migration to Postgres if not done.
  • Background job queue + SSE progress streaming.
  • Audit log for admin actions; admin console v2.
  • Privacy policy, ToS, account deletion, data export (GDPR).
  • OpenAPI generated from Zod; shared @gvc/contracts package.

6 Differentiate

Goal: features that no competitor has.
  • Asset Studio (regenerate / swap / upload assets per game).
  • Prompt Library + featured games.
  • Versioned snapshots / undo.
  • Realtime co-edit (later, big lift).
  • Studio plan with public API + CLI.
