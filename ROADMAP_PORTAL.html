Architecture Reset 2026

זה העדכון הכי חשוב במסמך: התוכנית כבר לא צריכה להיות "בוא נבנה הכל מסביב ל־Phaser/Template Builder", אלא "בוא נאחד את מה שכבר בנוי סביב GAME_ENGINE, Supabase ו־agent workflow אמיתי".

Supabase core Trigger.dev jobs GAME_ENGINE-first No free runtime code

החלטה חדשה: Supabase כמרכז הנתונים והמוצר

  • Supabase Postgres יהיה מקור האמת ל־users, games, snapshots, jobs, prompts, credit ledger, analytics ו־assets metadata.
  • Supabase Auth יחליף בהדרגה את custom JWT/password logic שקיים היום ב־backend.
  • Supabase Storage ישמש ל־assets, thumbnails ו־exports עם RLS ו־signed access.

גבול חשוב: Supabase לא יהיה orchestration engine

  • Edge Functions טובות ל־webhooks, signed URL helpers, Stripe callbacks ופעולות קצרות.
  • generation jobs ארוכים, retries מורכבים ו־multi-agent workflow ירוצו ב־Trigger.dev.
  • Temporal נשאר אופציה עתידית ל־Scale/Enterprise, לא בחירה נכונה לשלב הראשון.
דעתי על Supabase: זו בחירה טובה מאוד לפרויקט שלך עכשיו. היא חוסכת זמן על Auth, DB, Storage ו־RLS, והיא עדיין נשארת PostgreSQL אמיתי. החלק שבו צריך להיזהר הוא לא לדחוף לתוכה את כל ה־agent runtime. Supabase הוא הבית של הנתונים; Trigger.dev הוא פס הייצור של העבודה הארוכה.

Supabase: Good Decision, With Boundaries

הטבלה הזו מחליפה את ההמלצה הקודמת של "PostgreSQL + R2 + Cloud Run" כבחירת ברירת מחדל ל־MVP. לא כי ההמלצה ההיא גרועה, אלא כי Supabase נותן לך יותר מהירות פיתוח ופחות חלקים לתחזק.

אזור להשתמש ב־Supabase? החלטה מעשית
Database כן Supabase Postgres עם JSONB, indexes, RLS ו־migrations.
Auth כן Supabase Auth במקום custom JWT/password logic, עם profiles table משלימה.
Storage כן Private buckets ל־assets/exports, public רק אחרי publish gate.
Edge Functions כן, אבל מעט Stripe webhooks, signed URL helpers, lightweight API glue, לא agent jobs ארוכים.
Realtime כן, בזהירות status updates ל־generation jobs, אבל לא game multiplayer בזמן אמת ב־MVP.
Orchestration לא Trigger.dev יריץ את ה־workflow הארוך עם steps, retries ו־run status.

מה כבר הכנו עד עכשיו

זו תמונת מצב שבנויה מתוך הקוד הקיים בריפו, לא רק מתוך התכנון. הכוונה כאן היא להפריד בין "יש מסמך" לבין "יש מערכת שעובדת בפועל".

קיים קיים חלקית עדיין חסר

GAME_ENGINE קיים ובנוי נכון כבסיס בטוח

  • יש runtime דקלרטיבי עם GameDefinition מאומת.
  • יש ECS, Scene lifecycle, physics sync, camera, UI, audio ו־spawner systems.
  • יש ולידציה זהה כמעט בין ה־engine לבין backend.
  • יש build, typecheck ו־tests ברמת המנוע.

Backend של MVP כבר קיים, אבל עדיין מקומי מדי

  • יש auth, games CRUD, tokens, analytics, stats ו־downloads.
  • יש routes ל־MCQ generation, full game generation ו־game editing.
  • יש server-owned prompts ו־retry loops עם validation.
  • יש endpoint חדש ל־engine generation עבור ה־GameDefinition החדש.
  • חשוב: שכבת הנתונים הנוכחית מבוססת SQLite מקומי, לא Supabase/Postgres.

Frontend מודרני כבר קיים

  • React + TypeScript + routes מוגנים.
  • מסך builder עם prompt, MCQ, generation ו־preview.
  • מסכי login, dashboard, games list, game detail ו־stats.
  • תצוגת preview דרך iframe sandbox.

יש כיוון Engine-first, אבל החיבור עוד לא שלם

  • קיים route של /api/engine/generate.
  • ה־UI הראשי עדיין מחובר בעיקר ל־flow הישן של MCQ → JSON ישן → HTML template.
  • יש דו-קיום בין ארכיטקטורה ישנה לארכיטקטורה חדשה, וזה מייצר פיצול.
  • היעד החדש הוא להפוך את GameDefinition לפורמט היחיד של generation חדש.
מסקנה חשובה: לא מתחילים מאפס. יש אצלך כבר שלד חזק מאוד: מנוע ריצה, backend עובד, frontend עובד, ולידציה, auth וטוקנים. מה שחסר הוא להפוך את זה ממערכת generation "חד-פעמית" למערכת agentic מסודרת ורציפה סביב ה־GAME_ENGINE החדש, ולהחליף את שכבת הנתונים המקומית ב־Supabase בצורה מדורגת ולא שבירה.

מה עוד נשאר להכין ולייצר

אלו הפערים המעשיים בין מצב הקוד הנוכחי לבין החזון של הפלטפורמה כפי שמתואר במסמך.

1. לאחד את ה־product flow סביב המנוע החדש

  • להעביר את ה־builder הראשי לעבוד מול GameDefinition ולא מול הפורמט הישן בלבד.
  • להחליט אם flow הישן נשאר fallback או נכנס להוצאה משימוש מסודרת.
  • להציג preview מבוסס engine runtime ולא רק srcdoc template.

2. להקים שכבת Agents אמיתית עם Trigger.dev

  • Intake & Scope Agent.
  • Game Design & Mechanics Agent.
  • Runtime Blueprint & Asset Plan Agent.
  • QA, Patch & Publish Agent.
  • Trigger.dev orchestrator דטרמיניסטי שומר state ב־Supabase ומפעיל validators.

3. Supabase migration + Versioning

  • להעביר users/games/prompts/tokens/analytics מ־SQLite ל־Supabase Postgres.
  • לעבור מ־"גרסה אחרונה בלבד" ל־snapshots + diffs + last-good snapshot.
  • לאפשר rollback בטוח אחרי generation/edit כושל.
  • לשמור היסטוריית שינויים ברמת blueprint ולא רק prompt text.

4. Asset pipeline מאובטח ומלא

  • allowlists, MIME/signature validation, scan status ו־provenance.
  • storage פרטי + signed URLs בזמן ריצה בלבד.
  • packs פנימיים, thumbnails, exports ו־CDN strategy.

5. Preview sandbox ו־publish gates

  • engine-native preview שמריץ GAME_ENGINE ולא HTML template ישן בלבד.
  • preview domain מבודד.
  • CSP / iframe sandbox / SRI / asset restrictions.
  • smoke tests, visual diff ו־last-good publish only.

6. Billing ו־analytics ברמה מסחרית

  • יש token model בסיסי.
  • חסר credit UX, refund policy, usage settlement ו־overage logic.
  • חסרות metric dashboards ממוקדות business funnel.
  • היעד: credit_ledger ב־Supabase + Stripe Billing + refund על failures מערכתיים.

תקציר מנהלים

גרסה מסודרת של לב המסמך המקורי, עם שפה ברורה יותר לעבודה שוטפת.

כיוון המוצר והטכנולוגיה

ההחלטה המרכזית במסמך היא ש־Gaming Vibe Coding צריכה להיות פלטפורמת 3D-first. המשמעות היא ש־Phaser אינו צריך להיות מנוע המשחק הראשי, אלא מנוע משלים עבור HUD, UI, מסכי onboarding ותבניות 2D צרות. מנוע הריצה הראשי צריך להיות Three.js, עם Rapier לפיזיקה.

מבחינת AI, הכיוון המומלץ הוא data-first: המודל מחזיר JSON blueprints עם mechanics מתוך registry סגור, ולא code snippets חופשיים. זה מייצר אמינות, בדיקות, עלויות נשלטות ויכולת rollbacks.

מבנה Agents מומלץ

המסמך המעודכן ממליץ לעבוד עם ארבעה agents בלבד, אבל בסט חדש ומדויק יותר:

  1. Intake & Scope Agent - מבין את הרעיון, יוצר MCQ וקובע scope.
  2. Game Design & Mechanics Agent - מתכנן gameplay ומכניקות מתוך registry סגור.
  3. Runtime Blueprint & Asset Plan Agent - מייצר GameDefinition ותוכנית assets ראשונית.
  4. QA, Patch & Publish Agent - מאמת, מתקן ומחליט אם snapshot יכול להתפרסם.

סביבם נדרשת שכבת orchestration דטרמיניסטית ב־Trigger.dev, בסיס נתונים ב־Supabase Postgres, Storage מאובטח, sandbox קשיח ו־observability שמתחיל פשוט ומתרחב בהמשך.

המלצות תפעוליות ועסקיות

מבחינת תמחור, המסמך ממליץ על freemium + subscriptions + overage, כאשר המשתמש רואה credits ואילו המערכת הפנימית מתרגמת אותם לצריכת tokens.

מבחינת השקה, אסור למכור "כל משחק מכל סוג" ב־MVP. עדיף למכור template families, share/remix/embed מהרגע הראשון, ולמדוד first playable rate, edit success rate, share rate ו־cost per generation מהיום הראשון.

סטאק מומלץ חדש

זו ההמלצה המעודכנת שלי אחרי בדיקה מחדש של מצב הפרויקט. היא לא מנסה לבנות "הכי מושלם", אלא את המסלול הכי נכון כדי להגיע ל־MVP אמיתי בלי למחוק את מה שכבר עובד.

שכבה המלצה למה
UI React + Vite + TypeScript עכשיו ה־frontend כבר קיים ועובד. Next.js יישאר ל־marketing site עתידי, לא migration דחוף.
API Node + Express עכשיו, TypeScript בהדרגה Express כבר קיים. עדיף לחבר Supabase/agents/engine לפני החלפת framework.
DB / Auth / Storage Supabase Postgres, Auth, RLS, Storage ו־signed access תחת מוצר אחד שמתאים מאוד ל־MVP.
Agent Jobs Trigger.dev long-running AI tasks, retries, queues, realtime run status ופחות עומס תפעולי מ־Temporal.

Payments Stripe Subscriptions, credits, usage settlement ו־refund policy מסודר.
Observability Supabase logs + Trigger.dev dashboard, אחר כך Sentry/OpenTelemetry מספיק ל־MVP, עם נתיב ברור להרחבה כשיש משתמשים אמיתיים.

4 Agents החדשים

הבחירה הישנה של agents הייתה קרובה, אבל לא מספיק מדויקת לפרויקט הזה. הסט החדש מחלק אחריות לפי זרימת מוצר אמיתית, ומאפשר ל־agent אחד לקבל שתי מטלות כשזה חוסך פיצול מיותר.

1. Intake & Scope Agent

  • מבין את רעיון המשתמש.
  • יוצר MCQ רלוונטי.
  • קובע scope, genre, 2D/3D, difficulty ו־budget caps.
  • מונע הבטחות של "כל משחק מכל סוג" כבר בשלב הראשון.

2. Game Design & Mechanics Agent

  • מתכנן את לולאת המשחק והתחושה הכללית.
  • בוחר mechanics מתוך registry סגור.
  • מגדיר camera profile, controls, win/lose rules ו־difficulty curve.
  • אסור לו לייצר runtime code חופשי.

3. Runtime Blueprint & Asset Plan Agent

  • מייצר GAME_ENGINE GameDefinition תקני.
  • מכין asset plan ראשוני: placeholders, tags, provenance וסוגי נכסים.
  • חוסך agent נפרד ל־assets בשלב MVP.
  • מחזיר structured output בלבד.

4. QA, Patch & Publish Agent

  • קורא תוצאות schema validation ו־semantic validation.
  • מנתח smoke tests, playability checks ו־visual checks.
  • יוצר patch מתקן כשאפשר.
  • מחליט אם snapshot יכול להפוך ל־last_good.
חשוב: ה־orchestrator עצמו לא יהיה agent. הוא קוד דטרמיניסטי ב־Trigger.dev: מריץ שלבים, שומר state ב־Supabase, מפעיל validators, מחליט מתי לקרוא לכל agent ומתי לעצור.
User prompt → Intake & Scope → MCQ → Game Design → Blueprint + Assets → Validators → QA/Patch → last_good snapshot

Supabase Data Model

זה מודל הנתונים החדש שצריך להחליף את SQLite המקומי ואת שמירת "גרסה אחרונה בלבד".

טבלה תפקיד הערה חשובה
profiles פרופיל מוצרי משלים ל־Supabase Auth. לא לשכפל secrets או password data.
projects Workspace או container למשחקים של משתמש/צוות. בסיס ל־org/team בהמשך.
games metadata של משחק, owner, publish state. לא לשמור כאן כל snapshot מלא.
game_snapshots GameDefinition מלא, status, validation report. כולל working, last_good, published.
generation_jobs מצב job, progress, errors, model routing. Trigger.dev מעדכן את הטבלה בשלבים.
agent_runs קלט/פלט, model, latency, token cost לכל agent. בסיס ל־debugging ול־cost optimization.
prompt_events prompt מקורי, MCQ, answers, edit prompts. לא לערבב עם snapshots עצמם.
credit_ledger חיובים, holds, refunds ו־overage. ledger append-only ככל האפשר.
asset_objects metadata של assets, provenance, license, scan status. הקובץ עצמו יושב ב־Supabase Storage.
analytics_events first playable, edit success, publish/share, runtime metrics. להפריד product analytics מ־agent tracing.

MVP Phases החדשים

זו מפת דרכים מעודכנת שמחליפה את החלוקה הישנה. היא מתחשבת בזה שכבר יש backend, frontend ומנוע, ולכן לא מתחילים מ־"setup" אלא מאיחוד והקשחה.

Phase 1

Supabase foundation

להקים Supabase project, להעביר schema בסיסי, לחבר Auth/Storage, ולהגדיר RLS ראשוני.

Phase 2

Engine-native generation flow

להפוך את GameDefinition לפורמט הראשי של generation, ולחבר את frontend builder ל־engine route.

Phase 3

Trigger.dev + 4 agents

להוציא generation ארוך מה־request/response flow ולהריץ אותו כ־job עם progress, retries ו־agent runs.

Phase 4

Preview, validation, snapshots

להוסיף snapshots, last-good, validators, smoke tests ו־engine-native preview sandbox.

Phase 5

Credits, Stripe, share/remix

להפוך token system פנימי ל־credit UX מסחרי, עם refunds, plans, share links ו־remix.

Phase 6

Beta hardening

להקשיח security, observability, asset policy, device performance ו־support workflows לפני פתיחה רחבה.

Decision Ledger

אזור קצר שמכריח אותנו להיות כנים: מה שומרים, מה משנים, מה דוחים ומה מפסיקים לעשות.

Keep
React/Vite, Express כרגע, GAME_ENGINE, JSON-first, iframe preview, server-owned prompts, token tracking בסיסי.
Change
SQLite → Supabase, custom JWT → Supabase Auth, old JSON format → GameDefinition, one-shot generation → Trigger.dev jobs.
Defer
Temporal, full Next.js migration, marketplace, open-source model hosting, advanced 3D asset generation, multiplayer.
Drop
generated runtime code snippets, single latest version only, charging users for system-failed generations, trying to support every genre in MVP.

עקרונות ארכיטקטורה

הנקודות שצריך לזכור כל פעם שמקבלים החלטה חדשה בפרויקט.

עיקרון 1: להפריד בין Authoring ל־Runtime

ה־AI מייצר specification, לא קוד חופשי. ה־runtime שלך הוא זה שמממש את ה־spec דרך מערכות פנימיות בטוחות.

עיקרון 2: Mechanics registry סגור

כל mechanic הוא מזהה ידוע עם params. כך ה־Template Builder או ה־engine runtime יודעים בדיוק מה מותר ומה אסור.

עיקרון 3: Sandbox ו־publish gates

preview לא יוצא החוצה לפני validation, smoke tests, asset checks ו־last-good snapshot.

עיקרון 4: מוצר מסחרי ולא רק דמו טכנולוגי

credits, refunds, share/remix, analytics ו־retention חשובים לא פחות מהמנוע עצמו.

מודל ה־Blueprint

זה לב המערכת: שפת המשחק שלך. ככל שהיא תהיה ברורה ויציבה יותר, כל שאר המערכת תהיה אמינה יותר.

שדה חובה תפקיד
schemaVersion כן גרסאות של החוזה ושל ה־runtime.
metadata כן שם, ז'אנר, קהל יעד ו־UX metadata.
runtime כן בחירת מנוע ראשי, overlay ופיזיקה.
world כן פרמטרי העולם, dimension, boundaries וסקייל.
camera כן camera profile והתנהגות תנועה.
scenes כן יחידות gameplay, menu, HUD ו־results.
entities כן שחקן, אויבים, pickups, UI actors ועוד.
mechanics כן רשימת mechanics מתוך registry סגור.
assets כן נכסים עם provenance, license ו־scan status.
rules כן חוקי ניצחון, הפסד, timers, score ו־flows.
הערה חשובה: במסמך שלך זה מתואר יפה ברמה הקונספטואלית, ובקוד עצמו כבר יש התחלה טובה מאוד בכיוון הזה בתוך GAME_ENGINE. זה אומר שהמסמך והקוד כבר לא סותרים אחד את השני, אלא רק צריכים יישור קו מלא.

אבטחה, Sandbox ו־Uploads

האזור שהכי קל לדחות, אבל הכי יקר לשלם עליו אחר כך.

מה חייבים לעשות
  • לא להריץ eval, new Function או dynamic snippets מתוך JSON.
  • לא לשמור secrets, signed URLs, API keys או credentials בתוך blueprints.
  • לבודד preview בתוך iframe sandbox עם CSP קשיח.
  • לשמור assets ב־storage פרטי ולעשות mapping ל־signed URLs רק בזמן ריצה.
  • לסרוק uploads, לשמור provenance ולהחיל allowlists אמיתיים.
מה לא לעשות
  • לא למשוך scripts חיצוניים ב־runtime בלי שליטה מלאה.
  • לא לפרסם preview ציבורי לפני smoke tests ו־visual diff.
  • לא להסתמך על system prompt validation כאבטחה.
  • לא לאפשר user uploads לא סרוקים או assets בלי metadata של license.

מודל עסקי ושיווק

החלק העסקי של המסמך המקורי חזק, וריכזתי אותו כאן בצורה יותר נגישה.

מודל מסחרי

  • Freemium + subscriptions + overage.
  • Credits כלפי המשתמש, token economics פנימי מאחורי הקלעים.
  • Refund policy הוגנת: לחייב רק על playable success.
  • Edit prompts זולים יותר מ־new generations.

מנוע צמיחה

  • Share link, remix button ו־embed מהיום הראשון.
  • פנייה ל־creators, marketers, education ו־indie prototyping.
  • מיצוב: "Turn an idea into a playable browser game in minutes".
תוכנית מחיר חודשי מה כלול למי זה מתאים
Free $0 5 credits, watermark, public only, storage קטן onboarding ו־UGC funnel
Creator $19 60 credits, private drafts, exports בסיסיים יוצרים בודדים
Pro $49 220 credits, faster queue, team-lite power users וסוכנויות קטנות
Studio $199 1000 credits, seats, audit logs, branded embeds חברות וסטודיואים

Checklist לפני השקה

קיצרתי את רשימות הבדיקה המקוריות לגרסה שקל לעבוד איתה בזמן אמת.

טכני

  • אין שום code snippets חופשיים ב־JSON.
  • כל agent עובד מול schema version קבוע.
  • last-good snapshot נשמר תמיד.
  • כל template עובר smoke test ו־visual baseline.

אבטחה

  • preview בתוך iframe sandbox עם CSP קשיח.
  • uploads עם allowlist, scan ו־provenance.
  • exports לא מכילים secrets או signed URLs.

מוצר ו־Billing

  • Free tier מוגבל היטב בקונקרנציה וב־storage.
  • refund על failures מערכתיים.
  • share/remix/embed flow זמין.

תפעול

  • observability מלא ל־generation, build ו־runtime.
  • metric dashboards סביב playable success ו־edit success.
  • publish gate שמונע הפצת outputs שבורים.

מפת פערים מול הקוד הקיים

החלק הזה נוסף במיוחד בשבילך: הוא מחבר בין המסמך לבין מצב העבודה שלך ממש עכשיו.

נושא מה יש היום מה עדיין חסר
Runtime בטוח יש GAME_ENGINE דקלרטיבי עם validation וטעינה מבוקרת. לחבר אותו כ־default runtime של המוצר כולו.
Generation flow יש MCQ, generation, edit ו־retry loops. להמיר את ה־flow הראשי ל־engine-native blueprints שרצים כ־Trigger.dev jobs.
Frontend builder יש UI שלם יחסית עם auth ו־preview. לשנות אותו לעבוד על engine flow החדש ועל versioned snapshots.
Agents יש generation חד-שלבי עם retries. אין עדיין 4-agent workflow עם Trigger.dev, agent_runs ו־job state ב־Supabase.
Database יש SQLite מקומי דרך better-sqlite3. צריך Supabase Postgres עם RLS, profiles, snapshots, jobs ו־credit ledger.
Versioning יש שמירת game records ו־prompt history בסיסי. חסרים snapshots, patches, rollback ו־last-good publish.
Assets יש asset manifest בסיסי. חסרים scan pipeline, provenance policy ו־private object storage מלא.
Business layer יש tokens, auth, analytics ראשוני ו־stats. חסרים credits UX, refunds, billing strategy ו־growth mechanics.
